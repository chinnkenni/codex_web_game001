# 《旧城回身》多 Agent 协作手册

这份手册的目标是让 Keni 只需要向“游戏开发组长”传达意图；组长负责拆解、并行调度、验收和汇总，不需要 Keni 分别追多个对话。

## 当前线程地图

- 总方案/游戏本体线：负责轻量 Web 游戏总体方案、正式游戏逻辑、线索系统、构建与本地预览。
- 道具分配线：已经重构道具前后依赖，补上 `spend` 前置、`needFlags`、关键道具保护和 `tools/audit-story-flow.mjs`。
- 剧情后台线：负责 `story-dev.html` 审阅后台、剧情节点、结局路线、道具流转表和后台阅读体验。
- 音效/发布线：可作为专项线程读取经验，但默认不直接作为当前开发主线。

## 总控职责

游戏开发组长负责：

- 接收 Keni 的自然语言目标，提炼硬性要求、范围和验收标准。
- 判断哪些任务可以并行，哪些必须串行集成。
- 给子 Agent 下发明确任务，限定文件范围和交付格式。
- 保护当前工作区，不回滚他人改动。
- 汇总子 Agent 结果，统一修改共享文件。
- 运行校验、构建、浏览器验证或手机预览。
- 最终只向 Keni 汇报结论、风险和下一步。

## Agent 角色

| 角色 | 适合任务 | 主要文件/产物 | 注意事项 |
| --- | --- | --- | --- |
| 游戏本体 Agent | 行动牌运行时、道具锁定、结局判定、线索系统、玩家体验 | `src/main.js`, `src/styles.css`, `index.html` | `src/main.js` 是高冲突文件，通常由组长最终集成 |
| 剧情文案/道具 Agent | 节点文案、道具来源用途、旗标命名、结局路线草案 | 方案、表格、或 `src/story-dev-rewrite.js` 补丁 | 必须说明每个门槛的前置来源 |
| 剧情后台 Agent | 审阅后台信息架构、路线审批、资料库、后台样式 | `story-dev.html`, `src/story-dev-rewrite.js`, `src/story-dev.css` | 后台不是孤岛，所有数据要能回流正式游戏 |
| 审计 QA Agent | 路线模拟、道具闭环、回归风险、验收清单 | `tools/audit-story-flow.mjs`, 测试报告 | 审计脚本不能被削弱来通过检查 |
| 视觉/音频 Agent | 回执图、音效、配音、素材命名和接入方案 | `assets/story/`, `assets/audio/`, 素材清单 | 资源接入后要更新引用和构建 |
| 发布/预览 Agent | README、版本号、构建产物、Cloudflare/手机预览 | `README.md`, `package.json`, `package-lock.json`, `index.html`, `story-dev.html`, `dist/`, `docs/phone-preview.md` | 手机预览遵守 `AGENTS.md` 的 Quick Tunnel 规则 |

## 默认协作流程

1. 组长读取当前状态：工作区、相关文件、必要时读取旧线程。
2. 组长拆任务：区分“探索”“内容设计”“代码实现”“验证发布”。
3. 并行派发：只把互不抢文件的任务交给子 Agent。
4. 组长本地推进关键路径：共享数据、`src/main.js`、最终整合通常不外包。
5. 回收结果：检查子 Agent 是否满足要求，必要时补改。
6. 统一验收：至少运行 `npm run check`；涉及发布或静态产物时运行 `npm run build`。
7. 可视验证：前端体验改动用本地浏览器看；手机审阅用 `npm run phone:preview`。
8. 汇报：告诉 Keni 做了什么、验证结果、仍需决定的点。

## 文件冲突规则

- 同一轮不要让两个 Agent 同时改 `src/main.js`。
- 剧情后台可以先改 `src/story-dev-rewrite.js`，但正式游戏适配必须随后同步到 `src/main.js`。
- `tools/audit-story-flow.mjs` 是闸门，不是绕路工具；新增规则可以加严，不能为了省事放松。
- `dist/` 只由构建生成，不手写。
- README、版本号、缓存参数和发布说明由组长或发布 Agent 收尾统一处理。
- 长期更稳的方向是抽出共享剧情数据源，让正式游戏和剧情后台都读取同一份结构化数据；在做到这一步之前，审计脚本就是防漂移闸门。

## 数据闭环硬规则

每次剧情、道具或结局变化，都必须回答：

- 玩家在哪里获得这个道具或旗标？
- 后面在哪里需要或消耗它？
- 如果行动有 `spend`，玩家没有该道具时是否会被锁住？
- 如果行动有 `needFlag` 或 `needFlags`，缺失时是否显示清楚原因？
- 四条结局路线是否仍能逐步走通：`旧王归位`、`以人离城`、`怪异出城`、`王权托付`？
- `npm run check` 是否通过 `Story flow audit passed.`？

## Keni 下指令模板

最省心的格式：

```text
目标：我要调整/新增……
重点：最在意……，不要破坏……
交付：只要方案 / 直接改代码 / 改完给本地预览 / 改完给手机预览
```

也可以只说一句自然语言，例如：

```text
把王权托付这条线再做得更隐蔽一点，但不要让道具链断掉。
```

组长会自动拆成：剧情文案方案、后台路线更新、正式游戏适配、审计验证。

## 子 Agent 派发模板

剧情文案/道具 Agent：

```text
你负责《旧城回身》的剧情与道具链方案。不要改正式游戏逻辑。
请围绕【目标】输出：节点变化、行动牌变化、道具 gain/need/spend、旗标 gain/needFlags、推荐结局路线影响、风险点。
```

剧情后台 Agent：

```text
你负责剧情开发后台。请只改后台相关文件，整理【目标】对应的节点、路线审批、资料库展示和道具流转。
完成后说明改动文件、后台如何审阅、哪些数据需要游戏本体同步。
```

游戏本体 Agent：

```text
你负责正式游戏适配。请根据剧情后台最新数据同步 MAP、CARD_LIBRARY、ITEMS、FLAG_REQUIREMENTS 和运行时锁定。
确保 need/spend/needFlag/needFlags 正确锁定，关键道具不被随机移除。
完成后运行 npm run check 和 npm run build。
```

审计 QA Agent：

```text
你负责只读或小范围审计。请验证本轮改动是否造成道具无来源、关键物无用途、门槛不可达、结局路线卡住、后台和正式游戏漂移。
优先给出可复现失败步骤和建议修复，不要削弱审计规则。
```

## 什么时候需要并行

适合并行：

- 新章节：文案设计、后台结构、正式游戏 UI、审计规则可以分开跑。
- 大型结局重构：每条结局路线可先独立推演，再由组长统一收束。
- 素材接入：素材清单/命名、代码引用、视觉验收可并行。

不适合并行：

- 小修一个 bug。
- 同时大量改 `src/main.js` 的行动牌数据。
- 审计失败原因还没定位时盲目多头修。
